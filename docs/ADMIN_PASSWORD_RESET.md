# دليل إعادة تعيين كلمة سرّ المدير

> دليل مستقلّ وآمن لإعادة تعيين كلمة سرّ أيّ مستخدم — وعادةً حساب `admin` —
> عندما لا تعود نقطة الكشف المرّة-الواحدة (one-time login banner) متاحة.

---

## 1. متى تُستخدم هذه العمليّة

استخدم هذا الإجراء **فقط** في إحدى الحالات التالية:

- **فقدان الكلمة الأوّليّة.** فاتك سطر طباعة كلمة المرور الأوّليّة في سجلّ التشغيل،
  أو تدوّر السجلّ بعد إعادة التشغيل، ونقطة `/api/initial-credentials` لم تعد متاحة
  (تُعيد `404`).

- **نسيان كلمة مرور حساب موجود** (مدير أو موظّف) ولا يوجد مدير آخر يستطيع إعادة
  تعيينها من واجهة الإعدادات.

- **أتمتة التهيئة.** تريد ضبط كلمة مؤقّتة معروفة في بيئة جديدة، ثم تُترك للمستخدم
  لتغييرها عند أوّل دخول.

> **لا تستخدم هذا الإجراء للتغييرات الاعتياديّة.** التغييرات اليوميّة تتمّ من خلال
> شاشة "تغيير كلمة المرور" بعد الدخول، أو من قِبَل مدير عبر صفحة الإعدادات.

---

## 2. ماذا يفعل السكربت

ينفّذ الملف `script/reset-admin-password.ts` تحديثاً واحداً (`UPDATE`) على صفّ
مستخدم واحد فقط، ولا شيء آخر:

| العمود                 | الإجراء                                                                  |
| ---------------------- | ------------------------------------------------------------------------ |
| `password`             | يُضبَط على بصمة `bcrypt` جديدة (cost = 10) لكلمة المرور الجديدة.            |
| `must_change_password` | يُجبَر على `true`، فيُطلَب من المستخدم تغيير الكلمة عند الدخول التالي.        |

كلّ ما عدا ذلك يبقى كما هو — `id`، `username`، `role`، وأيّ بيانات مرتبطة
(أعضاء، اشتراكات، …) **لا تُمسّ**.

---

## 3. الضمانات الأمنيّة

- **لا يُشغَّل تلقائيّاً مطلقاً.** يُستدعى يدويّاً من سطر الأوامر فقط.

- **لا تُكتب كلمة المرور في أيّ ملف ولا في أيّ سجلّ دائم.** الكلمة المُولَّدة تظهر
  في الطرفيّة مرّة واحدة فقط، ثم تُمحى من ذاكرة العمليّة.

- **فحص مسبق للوجود.** السكربت يقرأ المستخدم أوّلاً، وإن لم يوجد، يفشل قبل أيّ
  محاولة كتابة.

- **ضمان الذرّيّة.** يفشل السكربت إن لم يتأثّر بالتحديث صفّ واحد بالضبط.

- **تدوير إجباري.** خاصيّة `must_change_password` تُضبط دائماً على `true`،
  فالكلمة المُطبَّقة تُعامل كمؤقّتة دائماً.

- **حماية الإنتاج.** عندما تكون `NODE_ENV=production`، يرفض السكربت العمل ما
  لم تُمرَّر `CONFIRM=yes` بشكل صريح في البيئة.

---

## 4. المتطلّبات

| المتطلّب           | التفاصيل                                                                                                  |
| ------------------ | --------------------------------------------------------------------------------------------------------- |
| Node.js والحزم     | تمّ تشغيل `npm install` في النسخة الحاليّة (السكربت يستخدم `tsx`، `bcryptjs`، `drizzle-orm`).                |
| `DATABASE_URL`     | يجب أن يشير إلى قاعدة البيانات التي تريد تحديث صفّ المستخدم فيها.                                            |
| وصول إلى Shell     | وصول مباشر إلى Shell البيئة التي يكون فيها `DATABASE_URL` متاحاً (بيئة التطوير، Shell نشر Replit، إلخ).    |

> **تأكّد أنّ `DATABASE_URL` يشير إلى قاعدة البيانات الصحيحة.** الخطأ في البيئة
> هو أكبر خطر في هذا الإجراء — اقرأ الرابط مرّتين قبل التنفيذ على الإنتاج.

---

## 5. أوضاع الاستخدام

كلّ الأوامر تُنفَّذ من **مجلّد المشروع الجذري**. غيّر القيم حسب الحاجة.

### 5.1 توليد كلمة قويّة عشوائيّة لحساب `admin`

```bash
npx tsx script/reset-admin-password.ts
```

تظهر الكلمة الجديدة (24 حرفاً، أبجديّة آمنة لعناوين URL، بدون أحرف ملتبسة)
داخل إطار مميّز **مرّة واحدة فقط**. انسخها فوراً — لن يمكن استعادتها لاحقاً.

### 5.2 استخدام كلمة من اختيارك

عبر وسيط سطر الأوامر:

```bash
npx tsx script/reset-admin-password.ts --password='My$tr0ng!Pass'
```

أو — وهو الأفضل لأنّه لا يظهر في تاريخ الـShell — عبر متغيّر بيئة:

```bash
ADMIN_RESET_PASSWORD='My$tr0ng!Pass' npx tsx script/reset-admin-password.ts
```

يجب أن لا تقلّ الكلمة عن **8 أحرف**. السكربت لا يعيد طباعتها — يكتفي بتأكيد
أنّ القيمة المُمرَّرة طُبِّقت.

### 5.3 استهداف مستخدم آخر

```bash
npx tsx script/reset-admin-password.ts --user=other_admin
```

يمكن دمجها مع أيّ وضع من أوضاع كلمة المرور أعلاه:

```bash
ADMIN_RESET_PASSWORD='Temp!Pass2026' \
  npx tsx script/reset-admin-password.ts --user=jane
```

### 5.4 المساعدة

```bash
npx tsx script/reset-admin-password.ts --help
```

## 6. أوامر العمل في بيئة Replit عبر Shell

هذا القسم مخصّص لمن يعمل داخل Workspace أو Deployment على منصّة Replit،
ويوضّح خطوات التنفيذ من نافذة **Shell** مباشرةً.

> **ملاحظة جوهريّة:** في Replit يوجد نوعان من الـShell:
> - **Workspace Shell** — يرى قاعدة بيانات التطوير (نفس قاعدة بيانات بيئة العمل).
> - **Deployment Shell** — يرى قاعدة بيانات الإنتاج عبر `DATABASE_URL` الخاصّ بالنشر.
>
> كل أمر أدناه يُنفَّذ في الـShell **المناسبة** للبيئة التي تستهدفها. لا تُنفِّذ
> أمر إنتاج من Workspace Shell، ولا العكس.

### 6.1 فتح نافذة Shell

- **بيئة التطوير (Workspace):** افتح المشروع في Replit، ثمّ من القائمة الجانبيّة
  اختر **Tools → Shell** (أو اضغط على لسان "Shell" في الأسفل).

- **بيئة الإنتاج (Deployment):** افتح صفحة النشر (Deployments)، اختر النشر
  المعنيّ، ثمّ افتح لسان **Shell** الخاصّ به (وليس Shell بيئة التطوير).

### 6.2 خطوات التحضير قبل التنفيذ

1. التأكّد من المجلّد الحالي (يجب أن يكون جذر المشروع):

   ```bash
   pwd
   ls package.json
   ```

2. التحقّق أنّ `DATABASE_URL` مضبوط ومُسجّل لقاعدة البيانات الصحيحة، مع إخفاء
   كلمة المرور أثناء الطباعة:

   ```bash
   echo "$DATABASE_URL" | sed 's|//[^@]*@|//<redacted>@|'
   ```

3. (اختياري) فحص الاتّصال بقاعدة البيانات وعدد المستخدمين الموجودين:

   ```bash
   npx tsx -e "
   import { db } from './server/db.js';
   import { users } from './shared/schema.js';
   const rows = await db.select().from(users);
   console.log('عدد المستخدمين:', rows.length);
   console.log('الأسماء:', rows.map(r => r.username).join(', '));
   process.exit(0);
   "
   ```

   إن فشل، فالأرجح أنّ الحزم لم تُثبَّت — نفّذ `npm install` ثمّ أعد المحاولة.

### 6.3 إعادة تعيين كلمة `admin` في بيئة التطوير على Replit

من **Workspace Shell**:

```bash
npx tsx script/reset-admin-password.ts
```

ستظهر كلمة المرور الجديدة في إطار مميّز. انسخها فوراً، ثمّ افتح المعاينة (Webview)
وسجّل دخولك بها.

### 6.4 إعادة تعيين كلمة `admin` في بيئة الإنتاج على Replit

من **Deployment Shell** (فقط)، نفّذ مع تفعيل حماية الإنتاج:

```bash
CONFIRM=yes NODE_ENV=production \
  npx tsx script/reset-admin-password.ts
```

ثمّ انسخ الكلمة المؤقّتة المطبوعة، وسجّل الدخول مباشرةً كـ`admin` على رابط
النشر (`<اسم-التطبيق>.replit.app` أو نطاقك الخاصّ). سيُجبرك التطبيق على وضع
كلمة جديدة قبل أيّ إجراء آخر.

### 6.5 إعادة تعيين كلمة مستخدم غير `admin` على Replit

```bash
npx tsx script/reset-admin-password.ts --user=jane
```

أو في الإنتاج:

```bash
CONFIRM=yes NODE_ENV=production \
  ADMIN_RESET_PASSWORD='Temp!Pass2026' \
  npx tsx script/reset-admin-password.ts --user=jane
```

### 6.6 ملاحظات عمليّة خاصّة بـReplit

- **متغيّرات البيئة في Replit** تُدار من قائمة **Secrets** في الـWorkspace أو من
  إعدادات النشر للإنتاج. لا تكتب `DATABASE_URL` يدويّاً عند العمل على Replit؛
  فهو يُحقَن تلقائيّاً.

- **بعد إعادة التعيين**، يُستحسن إعادة تشغيل السيرفر (إيقاف وتشغيل الـWorkflow
  من زرّ التشغيل) ليأخذ أي تغيير في الجلسات أثره فوراً، رغم أنّ السكربت لا
  يتطلّب ذلك تقنيّاً.

- **سجلّات Shell** على Replit تُمحى بإغلاق النافذة، لكن لا تعتمد على ذلك —
  امسح ذاكرة الـ scrollback يدويّاً بعد أن تنسخ الكلمة (`clear`).

- **لا تستخدم** أمر `git commit` أو `git push` لتسجيل أيّ كلمة مرور أو ناتج
  للسكربت، حتى لو ظهر مؤقّتاً في الـShell.

## 7. التشغيل على الإنتاج (Replit Deployment)

1. افتح نشر التطبيق في Replit وابدأ **Shell** الخاصّة بالنشر (وليس Shell
   بيئة التطوير) — هذه فقط هي التي ترى `DATABASE_URL` الإنتاجي.

2. تأكّد أنّك على قاعدة البيانات الصحيحة:

   ```bash
   echo "$DATABASE_URL" | sed 's|//[^@]*@|//<redacted>@|'
   ```

3. نفّذ مع حماية الإنتاج:

   ```bash
   CONFIRM=yes NODE_ENV=production \
     npx tsx script/reset-admin-password.ts
   ```

4. انسخ الكلمة المؤقّتة المطبوعة، ثم سجّل الدخول مباشرةً كـ`admin`. سيُجبرك
   التطبيق على وضع كلمة جديدة قبل أيّ إجراء آخر.

5. **يُستحسن:** أن تغيّر الكلمة مرّة أخرى من واجهة التطبيق بعد الدخول؛ لا تترك
   القيمة المؤقّتة في الاستخدام.

## 8. الإخراج المتوقّع

عند النجاح يطبع السكربت إطاراً واضحاً:

```
╔══════════════════════════════════════════════════════════════╗
║  Password reset successful                                   ║
║  تم إعادة تعيين كلمة المرور بنجاح                              ║
╠══════════════════════════════════════════════════════════════╣
║  username: admin                                             ║
║  role:     admin                                             ║
║  password: <fresh value>                                     ║
╠══════════════════════════════════════════════════════════════╣
║  • Copy the password NOW — it will not be shown again.       ║
║  • The user MUST change this password on their next login.   ║
╚══════════════════════════════════════════════════════════════╝
```

إذا مرّرت كلمتك الخاصّة، فإنّ سطر `password:` يكتفي بعبارة "(القيمة التي
زوّدتها قد طُبِّقت)" دون ترديد السرّ.

## 9. معالجة المشكلات

| الرسالة                                                | المعنى                                                                       | الحلّ                                                                            |
| ------------------------------------------------------ | ---------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| `DATABASE_URL is not set.`                             | متغيّر البيئة غير موجود في الـShell الحاليّة.                                  | افتح Shell فيها المتغيّر مضبوطاً (مثل Deployment Shell)، أو صدّره يدويّاً.          |
| `Refusing to run in production without confirmation.`  | اكتُشف `NODE_ENV=production` بدون `CONFIRM=yes`.                              | أعد التنفيذ بصيغة `CONFIRM=yes NODE_ENV=production ...` بعد التأكّد.              |
| `User "<name>" was not found.`                         | لا يوجد صفّ في جدول `users` يطابق الاسم المُمرَّر عبر `--user`.                  | تأكّد من الإملاء، أو اعرض المستخدمين من صفحة الإعدادات أوّلاً.                       |
| `Provided password must be at least 8 characters.`     | القيمة الممرّرة عبر `--password` أو `ADMIN_RESET_PASSWORD` قصيرة جدّاً.          | اختر قيمة أطول، أو احذف الوسيط لتوليد كلمة قويّة تلقائيّاً.                          |
| `Update did not affect exactly one row (rows=...)`     | تنبيه أمان — التحديث لم يطابق صفّاً واحداً بالضبط.                              | افحص جدول `users` مباشرةً قبل أيّ إعادة تنفيذ.                                     |

## 10. توصية بالتدوين (Audit Log)

السكربت لا يتواصل مع أيّ خدمة خارجيّة، ولكن في الإنتاج يُستحسَن أن تحتفظ
بسجلّ يدوي خارج النظام يحوي:

- من نفّذ العمليّة.
- متى نفّذها (الوقت بدقّة).
- أيّ حساب أُعيد تعيينه.
- السبب.
- تأكيد أنّ المستخدم النهائي غيّر الكلمة المؤقّتة بعد دخوله.

## 11. ملفّات ذات صلة

- `server/storage.ts` — الدالّة `initializeAdmin()` هي المنطق الأصلي الذي
  ينشئ حساب `admin` على قاعدة بيانات جديدة ويطبع الإطار المرّة-الواحدة.
- `server/routes.ts` — يكشف نقطة `GET /api/initial-credentials` التي تُظهر
  الكلمة الأوّليّة لأوّل جلسة متصفّح، ثمّ تُمسحها نهائيّاً.
- `docs/SECURITY.md` — السياسة الأمنيّة الشاملة للمشروع.
